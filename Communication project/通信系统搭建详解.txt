版本九：客户端实现
ncat：第三方通用网络调试工具，不属于你的项目代码，只是临时 TCP 收发工具
client.go：你自己编写的业务客户端，属于项目源码，可以定制逻辑、对接你的聊天室协议
为什么要用自研客户端替代命令行 ncat?
（1）完全自定义交互逻辑，不受工具限制
ncat 只能原始收发字符串，无法封装快捷功能；你的 client 可以内置专属逻辑：
自动登录、上线自动改名
私聊快捷指令封装（比如输入对方昵称直接发起私聊，不用手打/to）
收到私聊高亮提醒、公屏消息格式美化
断线自动重连，ncat 断开就得手动重新输命令连接
举例：你后续可以写代码，检测到[私聊]开头内容就变色输出，ncat 做不到这种业务定制。
(2)贴合你自己的通信协议，无缝对接服务端规则
你的服务端是自定义协议：/rename、/list、/to 用户名 内容
ncat 是通用工具，不会识别你的协议语义，只是无脑转发字符
自研客户端可以做语法校验、本地提示：比如输入错误私聊格式，客户端本地直接提示，不用发到服务端才报错
(3)解耦外部工具依赖，环境通用性更强
使用 ncat：别人运行你的项目，电脑必须预装 ncat，Windows 默认不带，环境缺失就无法测试
使用 client.go：只要有 Go 环境，直接go run client.go就能运行，零外部工具依赖，项目完整性闭环。
(4)分层架构规范：标准 C/S 架构必须两端自研
你的项目是标准服务端 - 客户端架构：
server：业务后端（连接管理、消息转发、私聊、超时管控）
client：前端交互入口（用户输入、消息展示、本地交互）
只用 ncat 相当于没有自己的客户端层，架构残缺，课程作业 / 项目答辩里，完整两端代码才是合格成品，单纯命令行测试不算完整项目。
(5)后续扩展上限极高，ncat 完全无法拓展
基于 client.go 可以迭代完整客户端能力：
读取控制台键盘输入，封装发送函数
异步读写分离（一个协程收服务端消息，一个协程读键盘输入）
后期改成 GUI 图形客户端（绑定 GUI 库）、网页 WebSocket 客户端
ncat 永远只能停留在黑白命令行原始收发，无法业务迭代。
(6)学习层面：吃透 TCP 全流程
用 ncat 你只能看到收发结果，看不到客户端 TCP 连接、读写协程、连接断开处理的底层代码；
手写 client.go，你能完整掌握：net.Dial建立连接、conn.Read/Write流式读写、协程并发处理、连接异常捕获，是 Go 网络编程核心练习。

双方沟通全展示：
根据通信系统架构图，可知分为server和client两部分，server负责处理用户的请求，client负责展示用户的界面。
对于server部分，有一个OnlineMap，存储在线用户的信息，包括用户的id、用户本身。此外，还有一个Message部分，是一个channel，用来向所有用户广播消息，也能接收用户信息。
对于client部分，有多个client，每个都是一个用户，会使用用户类实例化每一个具体用户。并且，每个用户还有自己的user对象，用在自己的channel上，表示该用户使用自己的channel接收服务器的消息。
