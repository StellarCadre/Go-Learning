完整项目整体总结（结合你汇总文档梳理）
一、项目整体架构：标准 C/S TCP 聊天室
分为服务端 Server、客户端 Client两大模块，完全分离开发：
服务端（后端处理层）：负责连接管理、并发、消息分发、指令解析、用户状态维护
客户端（交互层）：负责控制台菜单、用户输入、封装协议、展示服务推送消息
二、服务端完整迭代链路（版本 1 ~ 版本 8）
V1 基础裸服务
搭建Server结构体、监听端口、基础 Accept 连接，仅能建立 TCP 连接，无用户概念。
V2 用户上线机制
新增user.go；Server 增加在线用户 map、读写锁、全局广播通道；新增上线广播，多客户端上线互相可见提示。
V3 全员消息广播
在连接处理协程增加持续读取客户端消息，普通消息自动广播给全部在线用户。
V4 业务封装重构
将上下线、消息处理逻辑从handleConn抽离封装进User方法：Online()/Offline()/DoMessage()，代码解耦易扩展。
V5 在线用户查询 /list
DoMessage增加指令分支，识别/list，单独将在线列表单发查询者，不全局广播。
V6 改名指令 /rename
DoMessage新增改名分支，校验空值、重名，同步修改在线 map 键，广播改名通知。
V7 300 秒空闲超时强踢
handleConn增加活跃通道 + select 超时逻辑，长时间无输入自动下线、关闭连接。
V8 私聊指令 /to
DoMessage增加私聊分支，分割目标昵称与内容，仅推送指定用户，区分公聊 / 私聊。
服务端核心设计亮点
并发安全：sync.RWMutex读写锁保护在线用户 map；
解耦广播：全局Message管道 + 独立ListenMessager协程统一分发；
用户私有通道：每个User单独 channel，实现点对点私聊；
分层处理：网络层handleConn、业务层User.DoMessage分离；
保活机制：空闲超时自动清理连接，避免无效连接占用资源。
三、客户端完整迭代链路（V1 ~ V6）
前期依赖telnet/ncat临时测试，后续自研完整client.go替代第三方工具，解决工具无自定义交互、依赖环境、无法封装协议的缺陷。
客户端 V1 基础连接客户端
硬编码 IP 端口，仅实现 TCP 拨号，无消息收发、无菜单，仅能挂住连接。
客户端 V2 flag 命令行参数
新增flag解析，支持默认127.0.0.1:8000，也可自定义-ip -port启动，支持-h查看帮助，TCP 连接逻辑不变。
客户端 V3 主菜单系统
搭建交互菜单（1 公聊 / 2 私聊 / 3 改名 / 0 退出），区分功能入口，完成输入数字校验，后台单独协程接收服务端推送消息。
客户端 V4 改名功能
实现Rename()，自动封装/rename xxx协议，支持exit放弃改名，空输入拦截。
客户端 V5 公聊功能
实现PublicChat()，循环发送全局消息，输入exit返回菜单，自动追加换行防止粘包。
客户端 V6 私聊功能
实现PrivateChat()，先输入目标昵称，循环发消息，自动封装/to 昵称 内容，非法输入拦截。
客户端核心设计亮点
收发分离：单独 goroutine 持续读取服务消息，不阻塞控制台输入；
统一输入方案：全程bufio.Scanner，规避Scan换行缓存污染；
协议封装：客户端自动拼接/rename//to指令，用户无需手动输入协议头；
友好交互：所有功能模块均支持输入exit快速退回主菜单；
灵活部署：命令行参数自定义服务端地址，适配本机 / 局域网连接。
四、自定义通信协议（服务端 DoMessage 统一解析）
/list：查询全部在线用户，单发本人；
/rename 新昵称：修改用户名，全服广播变更；
/to 目标昵称 消息：一对一私聊，仅收发双方可见；
无前缀纯文本：全局公聊，广播所有在线客户端。
五、项目运行规范
服务端启动命令
plaintext
go run main.go Server.go user.go
默认监听127.0.0.1:8000，如需局域网访问将监听 IP 改为0.0.0.0
2. 客户端三种启动方式
plaintext
# 默认本机
go run client.go
# 自定义局域网服务端
go run client.go -ip 192.168.1.100 -port 8000
# 查看参数说明
go run client.go -h
